[목표]
이 코드를 “실전 자동매매 운영 기준”으로 리뷰해줘.
단순 코드 품질이 아니라, 실제 돈이 걸린 환경에서 터질 수 있는 리스크를 중심으로 분석해줘.

특히 아래 상황을 기준으로 판단:
- 장중 실시간 데이터 처리
- 주문/체결/계좌 상태 불일치
- 프로세스 중단(크래시) 후 복구
- 백테스트 vs 실전 괴리

[최우선 검토 영역]
다음 항목을 반드시 집중적으로 검토:
1. 리밸런스 / 주문 흐름 중 “중간 상태에서 크래시 시 복구 가능 여부”
2. 상태 저장 타이밍 (checkpoint 위치 적절성)
3. 계좌 vs 내부 포트폴리오 sync mismatch 가능성
4. 실시간 데이터 처리 누락 / 이벤트 루프 블로킹
5. 체결 이벤트 중복 / 누락 / 순서 꼬임
6. stale data 사용 여부 (가격/포지션 timestamp)
7. race condition / 중복 실행 (특히 주문/체결)
8. 백테스트 vs 라이브 로직 불일치
9. 비용/슬리피지/TP/SL 기준값 일관성
10. 예외 발생 시 상태 보존 가능 여부

[출력 형식 - 반드시 준수]
1. 핵심 결론
   - 실전 사용 가능 여부 (가능 / 조건부 / 불가)
   - 가장 위험한 단일 포인트 (1개만 명확히)

2. 문제 요약 (Top 5, 심각도 포함)
   - P0 / P1 / P2로 구분
   - “실제 어떤 상황에서 터지는지”까지 설명

3. 원인 분석
   - 확정된 문제 vs 가설 구분
   - 코드 위치 (파일 + line 기준)

4. 수정 제안 (최소 수정 원칙)
   - 구조 변경 없이 해결 가능한 방법 우선
   - 필요 시 코드 단위 patch 제시

5. 검증 방법 (재현 가능한 시나리오)
   - 실제 운영 상황 기준 테스트
   - 크래시 / 네트워크 오류 / 부분체결 포함

6. 남는 리스크
   - 수정 후에도 남는 구조적 한계

[추가 요구]
- “괜찮아 보인다”보다 “망가질 수 있는 부분”을 우선적으로 찾아줘
- 로그 기준으로 실제 디버깅 가능한지 평가해줘
- silent failure는 반드시 지적해줘
- 상태 불일치 발생 가능성은 과하게 의심해도 좋음
